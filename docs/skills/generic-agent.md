# GenericAgent

## 快速判断

`GenericAgent` 不是普通意义上的单个 skill，也不包含可直接安装到本索引里的 skill 本体。它是一个自进化 Agent 框架，核心主张是：不要预置大量技能，而是让 Agent 在对话和任务执行过程中自动沉淀 SOP、生成技能树，并在后续任务中复用和迭代这些技能。

## 适合场景

- 你想研究 Agent 如何在长期使用中自我生成和迭代技能。
- 你希望理解“任务执行 → 反思 → 沉淀 SOP/Skill → 下次复用”的完整链路。
- 你关注低 token 开销、少量原子工具和分层记忆的 Agent 架构。
- 你需要一个可控制本地系统、浏览器、文件、终端和外部工具的 Agent 框架。
- 你想对比 Claude Code、OpenClaw 等 Agent 在工具数量、记忆系统、浏览器控制和 token 消耗上的差异。

## 前置要求

- Python 3.11 或 3.12。
- LLM API Key。
- 本地终端、文件系统和运行脚本的权限。
- 可选真实浏览器/桌面端环境，用于浏览器或系统控制。
- 愿意维护 Agent 生成的 SOP、技能树和长期记忆。

## 输入与输出

输入：

- 用户任务、目标或长程自动化需求
- 本地文件、网页、应用、终端或外部服务
- Agent 运行所需的模型、环境与权限配置

输出：

- 已完成的任务结果
- 可复用的 Skill/SOP
- 分层记忆和任务经验
- 个人化技能树
- 自动化脚本、工具调用链或环境配置

## 核心特色

- 元技能定位：不是单一任务 skill，而是关于“技能如何被 Agent 自己生成和迭代”的框架。
- 自进化机制：完成新任务后，把执行路径沉淀为可复用 Skill/SOP。
- 极简工具集：强调用少量原子工具组合出复杂系统控制能力。
- 分层记忆：通过不同层级保存规则、事实、任务技能和会话归档。
- 真实环境控制：支持本地系统、浏览器、终端、文件系统、键鼠和移动设备等能力扩展。
- Token 效率：项目 README 和 Linux.do 介绍都强调更低上下文开销和更高信息密度。

## 和相近技能的差异

- 相比本索引里的普通 skill，`GenericAgent` 不是为某个具体任务提供固定流程，而是提供让 Agent 自己长出流程的框架。
- 相比 [wechat-article-reader](wechat-article-reader.md)，它不是轻量脚本型工具；后者解决微信公众号读取，`GenericAgent` 关注如何把任意新任务沉淀成后续可复用能力。
- 相比 [karpathy-llm-wiki](karpathy-llm-wiki.md)，它不是维护某个主题知识库，而是维护 Agent 自己的分层记忆和技能树。
- 相比 [graphify](graphify.md)，它不以 Neo4j 图谱为核心产物，而以 Agent 执行循环、任务经验和 Skill/SOP 自进化为核心。

## 工作流程

1. 配置 Python、模型 API Key 和 GenericAgent 运行环境。
2. 通过桌面端、TUI 或其他前端启动 Agent。
3. Agent 用少量原子工具感知环境、读写文件、执行代码、控制浏览器或系统。
4. 面对新任务时自主探索、安装依赖、编写脚本、调试和验证。
5. 任务完成后通过反思机制，把有效执行路径沉淀为 Skill/SOP 或长期记忆。
6. 后续遇到相似任务时，从技能树和记忆中直接召回并复用。
7. 随使用时间增长，技能树逐渐变成个人化的 Agent 能力资产。

## 当前限制

- 这不是可直接安装的一条 skill，本索引将其作为“元技能/方法论”收录。
- 部署和使用门槛高于普通脚本型 skill，需要配置模型、运行环境和权限。
- 自生成技能的质量依赖任务验证、反思质量和后续维护。
- 涉及真实浏览器、桌面、文件系统和外部服务时，需要谨慎控制权限与安全边界。
- 更适合研究或构建长期 Agent 工作流，不适合作为单一任务的轻量工具。
