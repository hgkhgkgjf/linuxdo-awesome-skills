# agents-progressive-disclosure

## 快速判断

`agents-progressive-disclosure` 是一个用于治理 Agent 指令文件的 skill。它把过长的 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 或类似规则文件，重构成“精简入口文件 + 按需读取的专项文档/规则文件”，目标是在不丢规则的前提下减少常驻上下文负担。

## 适合场景

- 你的 `AGENTS.md` 或 `CLAUDE.md` 越写越长，重要规则被大量低频细节淹没。
- 你希望根指令文件像导航页，而不是百科全书。
- 你想把命令清单、搜索策略、包管理器规则、部署说明、框架细节和长示例拆到 docs。
- 你需要在重构前先扫描指令冲突，避免把相互矛盾的规则继续传播。
- 你希望后续 Agent 明确知道“遇到什么任务应该读哪份文档”。

## 前置要求

- 已有 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 或项目等价指令文件。
- 项目允许创建或修改 docs/rules 文件。
- 建议有 Git 或同目录备份，方便回滚。
- 遇到无法从上下文判断的规则冲突时，需要人工确认。

## 输入与输出

输入：

- 一个或多个 Agent 指令文件
- 现有 docs、rules、`.claude/`、`.cursor/rules/` 等规则结构
- 用户关于保留规则、拆分粒度或目标平台的要求

输出：

- 更短、更高信号的入口指令文件
- 按需读取的聚焦文档或规则文件
- 任务类型到文档的路由索引
- 规则保留检查结果
- 必要时的备份文件

## 核心特色

- 渐进式披露：高频规则常驻，低频细节按需读取。
- 路由入口：根文件负责告诉 Agent 什么时候读哪份文档。
- 先查冲突：移动规则前先扫描风格、命令、安全边界和优先级冲突。
- 保留意图：不是删规则，而是把规则放到更合适的位置。
- 兼容多平台：支持 `AGENTS.md`、`CLAUDE.md`、`GEMINI.md` 和平台专属规则目录。
- 验证闭环：用行数、关键词清单和最终复读确认规则没有丢失。

## 和相近技能的差异

- 相比 [GenericAgent](generic-agent.md)，它不是让 Agent 自我生长技能树，而是整理人为维护的 Agent 指令和规则文档。
- 相比 [deep-research](deep-research.md)，它不生成调研报告，核心产物是精简入口文件和按需文档。
- 相比 [karpathy-llm-wiki](karpathy-llm-wiki.md)，它不是主题知识库，而是项目协作规则和 Agent 工作约束的路由结构。
- 相比普通文档整理，它更强调安全规则、工具优先级、冲突扫描和规则保留验证。

## 工作流程

1. 确定目标指令文件，优先处理当前目录的 `AGENTS.md`，也支持用户指定的 `CLAUDE.md`、`GEMINI.md` 等文件。
2. 完整读取目标文件，并检查现有 docs/rules 结构。
3. 扫描相互矛盾的规则，例如代码风格、包管理器选择、测试命令、安全边界或优先级说明。
4. 把规则分为入口常驻规则和任务相关细节。
5. 设计 docs/rules 路由地图，优先复用项目已有命名和目录约定。
6. 创建备份后，把入口文件重写为紧凑路由器。
7. 将详细规则移动到聚焦文档，保留 frontmatter、imports、path globs 和文件扩展名。
8. 用关键词清单、行数和入口复读验证规则被保留，并确认入口能指导后续 Agent 按需读取。

## 当前限制

- 它处理的是 Agent 指令文件结构，不直接生成业务代码、报告或知识图谱。
- 重构质量依赖对原规则意图的理解，含糊或冲突规则可能需要人工确认。
- 安全、权限、工具优先级等关键规则不能只藏在按需文档里。
- 过度拆分会造成小文档泛滥，需要控制文档边界。
- 如果项目已有多套平台规则，需谨慎保留各自 frontmatter、路径匹配和导入机制。
